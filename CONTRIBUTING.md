# Contributing

## Testing

Before opening a PR, test the plugin locally on the surfaces you've changed. Follow the instructions below to test the plugin for every vendor supported:

- **Claude Code** — [Create your first plugin](https://code.claude.com/docs/en/plugins#create-your-first-plugin)
- **Cursor** — [Test plugins locally](https://cursor.com/docs/plugins#test-plugins-locally)
- **Codex** — [Install a local plugin manually](https://developers.openai.com/codex/plugins/build#install-a-local-plugin-manually)
- **Gemini CLI** — [Link your extension](https://geminicli.com/docs/extensions/writing-extensions/#step-4-link-your-extension)

## Releases

Most of this plugin's content is managed automatically. There are two distinct release concerns: the plugin itself, and the vendored agent skills.

### Plugin releases

Plugin versions are managed by [release-please](https://github.com/googleapis/release-please). When conventional commits land on `main`, release-please opens a release PR that bumps the version across all plugin manifests and MCP adapters. Merging that PR triggers a GitHub release and publishes the release archives.

The following files are bumped automatically on every release:

| File | Field |
|------|-------|
| `.release-please-manifest.json` | `.` (root version) |
| `.claude-plugin/plugin.json` | `version` |
| `.cursor-plugin/plugin.json` | `version` |
| `.codex-plugin/plugin.json` | `version` |
| `gemini-extension.json` | `version`, `mcpServers.supabase.headers.X-Source-Version` |
| `mcp/claude-code/.mcp.json` | `mcpServers.supabase.headers.X-Source-Version` |
| `mcp/cursor/mcp.json` | `supabase.headers.X-Source-Version` |

### Marketplace versions

`marketplace.json` files (`metadata.version`) are versioned independently from the plugin. They should only be bumped when a plugin is added to or removed from a marketplace entry — not on every plugin release.

When a PR modifies a `marketplace.json` file and the author hasn't already bumped `metadata.version`, the [`bump-marketplace-versions`](.github/workflows/bump-marketplace-versions.yml) workflow automatically commits a patch bump to the PR branch.

### Agent skills

The skills under `skills/` are vendored from [`supabase/agent-skills`](https://github.com/supabase/agent-skills) and must not be edited directly in this repo. Any change made here will be overwritten by the next sync.

#### How syncing works

1. When `supabase/agent-skills` publishes a new release, its CI dispatches the `sync-agent-skills` workflow in this repo automatically.
2. The workflow downloads every `.tar.gz` asset from that release, replaces the previously managed skill directories (tracked in `skills/.upstream.json`), and opens a PR.
3. As a fallback, the workflow also runs on a weekly schedule in case the dispatch was missed.

The PR is created using a GitHub App token so that CI (manifest validation) runs on it automatically.

#### Triggering a sync manually

Go to [Actions → Sync Supabase agent skills](https://github.com/supabase-community/supabase-plugin/actions/workflows/sync-agent-skills.yml) and click **Run workflow**. You can optionally supply a specific release tag from `supabase/agent-skills`; leave it blank to pull the latest release.

#### Tracking upstream state

`skills/.upstream.json` records the source release, version, commit SHA, and the list of managed paths and assets from the last sync. Do not edit this file manually.

#### Adding a new skill source

New skills published to `supabase/agent-skills` are picked up automatically — the sync workflow downloads all `.tar.gz` assets from each release without a hardcoded allowlist. No changes to this repo are needed when a new skill is added upstream.
